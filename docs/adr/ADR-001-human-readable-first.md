# ADR-001: Human Readable First

- Status: Accepted
- Date: 2026-07-02
- Related: PRD §3 First Principle, §9 Long-term Design Goals, [DATA_MODEL.md §8 Storage Layout](../DATA_MODEL.md#8-storage-layout-원칙-스케치)

## Decision

모든 데이터는 사람이 직접 읽을 수 있는 형태로 저장한다.

- 사진은 원본 파일 그대로 보존한다.
- 메시지, 메타데이터, 요약은 표준 텍스트 포맷(Markdown, JSON 등)으로 저장한다.
- 파일 시스템 구조 자체가 사람이 탐색 가능한 아카이브가 되도록 설계한다.
- 검색 인덱스, 캐시 등은 원본에서 언제든 재생성 가능한 파생 데이터로만 취급한다.

## Context

Duri는 수십 년 동안 사용할 것을 목표로 하는 두 사람의 Personal Life System이다. 앱보다 데이터가 오래 살아야 하며, 최악의 경우 Duri가 동작하지 않아도 사용자는 폴더를 열어 사진·메시지·메타데이터·요약을 직접 확인할 수 있어야 한다.

## Alternatives Considered

1. **DB 전용 저장 (RDB/NoSQL 내부 포맷에만 의존)** — 구현은 단순하지만 앱이 죽으면 데이터를 꺼내기 어렵고, 스키마 마이그레이션에 데이터 생존이 종속된다.
2. **클라우드 서비스(Firebase, Supabase Storage 등) 종속 저장** — 운영 부담은 줄지만 Vendor Lock-in이 발생하고 Ownership 원칙에 어긋난다.
3. **독점/바이너리 아카이브 포맷** — 저장 효율은 높지만 사람이 읽을 수 없고 장기 보존 리스크가 크다.

## Reason

- 데이터는 Duri만을 위한 것이 아니라 우리의 삶을 위한 자산이다.
- 표준 포맷 + 파일 시스템은 10년 뒤에도 어떤 도구로든 열 수 있다.
- Export가 별도 기능이 아니라 저장 구조 자체가 된다.
- AI·프레임워크·DB가 교체되어도 원본 데이터는 그대로 활용 가능하다.

## Consequences

- (+) 앱 없이도 데이터 접근이 가능하고, 백업·이전이 단순해진다.
- (+) 새 시스템으로의 마이그레이션 비용이 낮다.
- (−) DB 대비 저장 효율과 쿼리 성능에서 손해를 본다. 검색 성능은 파생 인덱스로 보완해야 한다.
- (−) 파일/디렉터리 구조 설계(DATA_MODEL.md)가 초기부터 중요해진다.
- 원본과 파생 데이터의 경계를 명확히 유지해야 한다(파생물은 언제든 삭제·재생성 가능해야 함).
